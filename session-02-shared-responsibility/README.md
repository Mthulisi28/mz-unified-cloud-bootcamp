\# ◆ SESSION 2 — SHARED RESPONSIBILITY + RISK AWARENESS



\## Objective

Understand how security responsibility is divided between AWS and the customer, and how misconfigurations create real cloud risks.



By the end of this session, students should understand:

\- AWS Shared Responsibility Model

\- What AWS secures vs what customers secure

\- IAM basics (Identity and Access Management)

\- Security Groups fundamentals

\- Common cloud security risks

\- Why governance is important early



\---



\# ◆ 1. SHARED RESPONSIBILITY MODEL



Cloud security is shared between:



\## AWS Responsibility

\- Physical datacenters

\- Hardware

\- Networking infrastructure

\- Global cloud infrastructure

\- Hypervisor layer



\## Customer Responsibility

\- Data

\- Identity \& Access (IAM)

\- Application configuration

\- Security settings

\- Network rules (Security Groups)



\---



\# ◆ 2. KEY PRINCIPLE



“AWS secures the cloud. You secure what you put in the cloud.”



\---



\# ◆ 3. IAM (IDENTITY \& ACCESS MANAGEMENT)



IAM controls:

\- Who can access AWS

\- What they can do

\- Which services they can use



\---



\# ◆ 4. SECURITY GROUPS



Security Groups act like virtual firewalls for EC2:



They control:

\- Inbound traffic

\- Outbound traffic

\- Port access rules



\---



\# ◆ 5. CLOUD RISKS



Common mistakes:

\- Public S3 buckets

\- Weak passwords

\- No MFA enabled

\- Over-permissioned IAM users

\- Opening all ports (0.0.0.0/0)



\---



\# ◆ 6. GOVERNANCE INTRODUCTION



Governance means:

\- Control

\- Structure

\- Security discipline

\- Visibility
